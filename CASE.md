# КЕЙС №1

## Автономное хранилище и сервис обработки записей звонков

**Кейсодержатель:** компания «Апогей»

---

## Описание

В компании используется телефония Mango Office, а записи звонков связаны с рабочими процессами и бизнес-сущностями в CRM. Однако сейчас записи фактически хранятся в личном кабинете Mango Office, где объём хранения ограничен, а старые записи со временем удаляются без возможности восстановления. Кроме того, из интерфейса CRM нельзя напрямую прослушать запись звонка или получить её расшифровку.

Ваша задача — разработать прототип автономного сервиса, который регулярно скачивает записи звонков из Mango Office, сохраняет их в структурированном и защищённом виде, ведёт журнал загрузок и предоставляет HTTP API для доступа к записям и связанным с ними данным. В дальнейшем этот сервис должен использоваться как инфраструктурная основа для CRM и будущих ИИ-сервисов: транскрибации, суммаризации, анализа качества консультаций, поиска проблемных звонков и других сценариев обработки.

На этапе хакатона полноценной интеграции с боевым контуром Mango Office и CRM может не быть, поэтому важно продумать автономную работу сервиса, подготовить модуль интеграции с Mango Office и показать работу на демонстрационных данных или тестовом контуре.

---

## Цель

Создать прототип автономного серверного сервиса, который:

- регулярно получает записи звонков из Mango Office;
- сохраняет аудиофайлы и метаданные звонков в собственном хранилище;
- предотвращает повторную загрузку дублей;
- предоставляет API для получения записи и связанных данных по ID звонка;
- ведёт журнал загрузок, ошибок и повторных попыток;
- формирует основу для последующей ИИ-обработки звонков;
- может быть развёрнут отдельно от CRM и взаимодействовать с ней по HTTP API.

Сервис должен быть самостоятельным компонентом корпоративной архитектуры. CRM в будущем будет обращаться к нему через API: например, чтобы получить запись звонка, расшифровку или результаты обработки. Интерфейс прослушивания записей со стороны CRM не входит в рамки задания.

---

## Описание проблемы

Записи звонков являются важным источником данных для техподдержки, продаж, контроля качества, руководителей, аналитиков и будущих ИИ-сервисов. Они позволяют восстанавливать контекст обращения, проверять качество консультаций, анализировать коммуникации с клиентами и находить повторяющиеся проблемы.

Сейчас записи звонков находятся в личном кабинете Mango Office.

Это создаёт несколько ограничений:

- старые записи могут удаляться из-за ограничения по месту хранения;
- нет единого автономного архива записей внутри инфраструктуры компании;
- CRM может хранить связь звонка с бизнес-сущностью, но не имеет удобного доступа к аудиозаписи;
- невозможно быстро построить сервисы расшифровки, анализа и поиска по содержанию звонков;
- нет единого журнала загрузок, ошибок и статусов обработки записей;
- нет подготовленного API, через которое CRM и будущие аналитические сервисы могли бы получать записи и результаты обработки.

В результате компания рискует терять важные данные, а развитие сервисов на основе анализа звонков затрудняется. Необходимо создать отдельный сервис, который решает задачу хранения и доступа к записям независимо от Mango Office и при этом готов к дальнейшему расширению.

---

## Требования к решению

### 1. Автономный сервис хранения записей

Необходимо реализовать отдельный HTTP-сервис с собственной базой данных и собственным хранилищем аудиофайлов.

Решение должно предусматривать:

- хранение записей звонков;
- хранение метаданных звонков;
- возможность получить запись через API;
- возможность получить связанные с записью данные через API;
- возможность дальнейшего расширения API без изменения общей архитектуры.

Хранение аудиофайлов является частью сервиса. Если команда использует внешнее объектное хранилище, файловое хранилище или аналог S3/minio, такое решение должно быть архитектурно обосновано.

---

### 2. Интеграция с Mango Office

Mango Office рассматривается как основной источник записей звонков. В перспективе могут появиться и другие источники, поэтому архитектура должна допускать расширение механизма загрузки.

На этапе хакатона интеграции с боевым контуром может не быть, но команда должна подготовить модуль интеграции, который в дальнейшем можно будет отладить и доработать при подключении к реальному API Mango Office.

Сервис должен:

- иметь фоновый механизм регулярной загрузки новых записей;
- получать и сохранять все доступные метаданные, которые предоставляет Mango Office;
- сохранять аудиофайлы записей;
- не создавать дубли при повторной загрузке;
- использовать уникальный ID звонка Mango Office как основной идентификатор для доступа к записи;
- корректно обрабатывать ошибки загрузки;
- повторять попытки скачивания, если запись не была загружена.

Команде рекомендуется изучить документацию Mango Office и учесть особенности API, ограничения, лимиты, форматы данных и механизм получения записей.

На заключительном этапе может быть предоставлен доступ к тестовому контуру Mango Office для проверки скачивания записей. До этого команда может использовать демонстрационные данные, заглушку API или заранее подготовленные тестовые записи.

---

### 3. API для CRM и внешних сервисов

Сервис должен предоставлять HTTP API, через которое CRM и другие внутренние сервисы смогут получать данные.

Минимально API должно позволять:

- получить информацию о записи по ID звонка;
- скачать аудиозапись по ID звонка;
- получить статус загрузки записи;
- получить расшифровку или результаты обработки, если они реализованы.

Авторизация API должна быть реализована по служебному ключу. Разграничение прав конечных пользователей будет выполняться на стороне CRM, поэтому в рамках задания не требуется реализовывать сложную пользовательскую модель доступа.

При проектировании API важно учитывать, что в будущем оно будет расширяться: появятся методы для получения транскриптов, результатов анализа, тегов, кратких содержаний и других данных.

---

### 4. Журналирование и административный дашборд

Пользовательский интерфейс для сотрудников, руководителей и операторов не требуется.

Основной пользовательский интерфейс в рамках хакатона — административный дашборд.

В дашборде желательно показать:

- общее количество сохранённых записей;
- количество успешно загруженных записей за период;
- количество ошибок загрузки;
- список последних операций загрузки;
- статус фоновой задачи;
- объём занятого и свободного места;
- базовую информацию о состоянии сервиса.

Также необходимо вести технические логи. Они должны помогать понять, какие записи были загружены, какие не удалось скачать и почему.

---

### 5. Безопасность и защищённое хранение

Структура и механизмы защиты остаются на усмотрение команды, но решение должно учитывать, что записи звонков содержат персональные данные и иную чувствительную информацию.

Будет преимуществом, если команда реализует или опишет:

- шифрование файлов или обоснование выбранного способа хранения;
- резервное копирование или подход к восстановлению данных;
- требования законодательства к системам, которые хранят подобные данные.

Обезличивание, маскирование номеров и сложная политика доступа конечных пользователей не являются обязательной частью задания, так как эти задачи могут решаться будущими сервисами и CRM.

---

### 6. Инфраструктура для ИИ-обработки

ИИ-обработка не является обязательной частью базового задания, но сервис должен быть спроектирован так, чтобы на его основе можно было строить следующие сервисы анализа звонков.

Потенциальные направления развития:

- расшифровка речи в текст;
- краткое содержание звонка;
- определение темы обращения.

Если команда реализует пример ИИ-обработки, это будет рассматриваться как бонус. Можно использовать локальные или внешние модели, если их применение обосновано и не нарушает ограничения соревнования.

Результаты обработки, если они реализованы, должны сохраняться рядом с записью в базе данных или связанном хранилище, чтобы их можно было быстро получить через API.

---

## Сценарии использования

### 1. Регулярная загрузка новых записей

Сервис по расписанию обращается к источнику Mango Office, получает список новых звонков и скачивает доступные записи. Для каждой записи сохраняются аудиофайл, ID звонка, дата, технические метаданные.

**Результат:** новая запись появляется в хранилище и становится доступной через API.

---

### 2. Защита от повторной загрузки

Сервис повторно получает из Mango Office информацию о звонке, который уже был загружен ранее. Система определяет, что запись уже есть в хранилище, и не создаёт дубль.

**Результат:** база данных и файловое хранилище остаются консистентными, повторная загрузка не приводит к дублированию данных.

---

### 3. Ошибка загрузки и повторная попытка

Во время скачивания записи возникает ошибка: файл временно недоступен, API Mango Office не отвечает или соединение обрывается. Сервис фиксирует ошибку в журнале, сохраняет статус неуспешной попытки и позже выполняет повторную загрузку.

**Результат:** запись не теряется, а администратор может увидеть проблему в логах или дашборде.

---

### 4. Регулярная загрузка новых записей

CRM или другой внутренний сервис обращается к API хранилища по ID звонка Mango Office. Сервис проверяет служебный ключ, находит запись и возвращает аудиофайл или ссылку/поток для скачивания в выбранной архитектуре.

**Результат:** CRM получает техническую возможность встроить прослушивание записи в свой интерфейс.

---

### 5. Защита от повторной загрузки

Администратор открывает дашборд и видит общее состояние хранилища: количество записей, объём занятого места, ошибки загрузки, последние операции и статус фонового процесса.

**Результат:** состояние сервиса можно контролировать без прямого обращения к базе данных или логам сервера.

---

### 6. Ошибка загрузки и повторная попытка

Администратор открывает дашборд и видит общее состояние хранилища: количество записей, объём занятого места, ошибки загрузки, последние операции и статус фонового процесса.

**Результат:** состояние сервиса можно контролировать без прямого обращения к базе данных или логам сервера.

## Функциональные требования

Формат решения: серверный HTTP-сервис с административным дашбордом.

Пользовательский интерфейс для конечных сотрудников не требуется.

Стек технологий: на усмотрение команды.

Сервис должен иметь собственную базу данных.

Сервис должен иметь собственный механизм хранения аудиозаписей или обоснованную интеграцию с отдельным хранилищем.

Взаимодействие с CRM должно проектироваться через HTTP API.

Архитектура должна быть модульной:

- загрузка из Mango Office;
- хранение;
- API;
- журналирование;
- административный мониторинг;
- будущая обработка.
